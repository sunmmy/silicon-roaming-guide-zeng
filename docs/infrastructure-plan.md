# 基础设施与同步方案

## 目标

这个网站后期必须做到低维护、低心智负担。理想状态是：

1. 在电脑上新增或修改一个项目内容。
2. 放入图片或二维码素材。
3. 执行一次同步。
4. 网站自动更新。

不希望出现的情况：

- 每次更新都要手动登录服务器。
- 每次都要改很多代码。
- 每次都要上传一堆零散文件。
- 需要自己维护 Linux 服务器、数据库、安全补丁。

## 推荐基础设施

### 第一阶段：轻量同步和快速上线

推荐组合：

- 本地：当前这个项目文件夹。
- 版本同步：Git + GitHub。
- 自动部署：Cloudflare Pages 或 Vercel。
- 域名：先不急，网站稳定后再绑定。

工作方式：

1. 本地改内容。
2. 推送到 GitHub。
3. 托管平台自动重新发布网站。

适合原因：

- 不需要买服务器。
- 不需要手动上传文件。
- 更新路径稳定。
- 以后换电脑也能拉回同一个项目。

### 第二阶段：正式域名和长期访问

当内容稳定后再做：

- 购买域名。
- 绑定到 Cloudflare Pages / Vercel。
- 如果特别重视国内访问，再考虑备案和迁移到国内静态托管 + CDN。

## 内容更新流程

后续应该把内容更新变成固定动作：

### 更新一个新项目

1. 在 `content/projects/` 新建一个项目 Markdown 文件。
2. 按模板填写项目名称、简介、背景、过程、成果和链接。
3. 把截图或封面放入 `assets/images/` 或 `public/images/`。
4. 本地预览确认没问题。
5. 同步到远程仓库，等待网站自动发布。

### 更新公众号二维码

1. 把二维码图片放入 `public/images/`。
2. 在关于/联系页引用它。
3. 本地预览。
4. 同步发布。

## 小白友好要求

后续实现时要尽量做到：

- 项目内容主要写 Markdown，不让用户频繁改代码。
- 图片只需要放到固定文件夹。
- 更新步骤尽量固定，不每次重新解释。
- 同步命令或按钮尽量少。
- 文档里保留“如何更新网站”的操作说明。

写项目内容 -> 放图片 -> 本地看一眼 -> 同步 -> 网站自动更新
